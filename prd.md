Проблема**

В бизнесе по организации праздников основной актив — инвентарь. При ручном учете возникают критические ошибки:

- Бронирование одного и того же реквизита на разные мероприятия в один период.
- Отсутствие понимания реального остатка на складе в конкретную будущую дату.
- Сложность в планировании закупок из\-за отсутствия статистики востребованности.

**1.2. Цели продукта**

- **Автоматизация контроля:** Система должна автоматически запрещать бронь, если свободного количества недостаточно.
- **Прозрачность:** Единая база данных с актуальными статусами всех предметов.
- **Оптимизация:** Минимизация времени на проверку доступности инвентаря при звонке клиента.

**1.3. Границы проекта (Scope)**

- **Включаем:** Учет количественных остатков, управление событиями, проверку пересечений дат.
- **Исключаем:** Финансовый учет (CRM), учет расходных материалов, личные кабинеты сотрудников (система для одного админа).

**Часть 2: Пользовательские истории и сценарии (User Stories)2.1. Управление каталогом (Inventory Management)**

- **Добавление позиций:** Я, как администратор, хочу добавлять новый инвентарь в систему (название, категория, количество), чтобы он стал доступен для бронирования.
- **Актуализация данных:** Я хочу редактировать общее количество предметов (например, при закупке новых или списании сломанных), чтобы данные на складе всегда были верными.
- **Удаление:** Я хочу иметь возможность удалять позиции, если они больше не используются, при условии, что они не забронированы в будущих мероприятиях.

**2.2. Планирование мероприятий (Event Planning)**

- **Создание и бронирование:** Я хочу создать карточку мероприятия с указанием дат, ввести необходимый инвентарь, и сразу же сохранить его в статусе «Забронировано», если инвентарь доступен.
- **Динамическая проверка:** Я хочу видеть реальный остаток каждого предмета **именно на выбранные даты**, чтобы не пообещать клиенту то, чего физически не будет на складе.
- **Бронирование:** Я хочу добавлять нужное количество предметов в заказ, чтобы система «закрепила» их за этим событием.

**2.3. Жизненный цикл и контроль (Lifecycle Control)**

- **Изменение дат:** Если клиент переносит праздник, я хочу изменить даты в системе, а система должна автоматически перепроверить, не занят ли уже этот инвентарь кем-то другим в новый период.
- **Завершение события:** Я хочу переводить мероприятие в статус «Завершено», чтобы задействованный в нем инвентарь снова стал доступен для других броней в будущем.

\-----**2.4. Основные сценарии (User Scenarios)**

Для архитектуры приложения мы выделим два главных сценария, которые ты будешь описывать в пояснительной записке:**Сценарий А: «Успешное бронирование»**

1. Админ создает событие «Свадьба» на 15.06 – 17.06.
2. Выбирает «Стул белый». Система показывает: _«Всего 50, доступно 40»_ (потому что 10 уже заняты другим заказом).
3. Админ вводит «30 шт.», нажимает «Добавить».
4. Система успешно сохраняет бронь.

**Сценарий Б: «Конфликт дат (Отказ)»**

1. Админ создает событие «Корпоратив» на те же даты 15.06 – 17.06.
2. Выбирает «Стул белый». Система показывает: _«Доступно 10»_ (так как 30 уже заняты «Свадьбой» и 10 другим заказом).
3. Админ пытается ввести «15 шт.».
4. **Результат:** Кнопка сохранения блокируется или выводится ошибка: _«Недостаточно свободного инвентаря на выбранный период»_.

\-----**2.5. Критерии приемки (Acceptance Criteria)**

Чтобы считать эту часть реализованной, система должна:

1. **Математическая точность:** Корректно суммировать все брони, пересекающиеся хотя бы на один час с выбранным интервалом.
2. **Реактивность:** Обновлять данные о доступности сразу после изменения дат мероприятия в интерфейсе.
3. **Целостность:** Не позволять сохранять изменения, если общее количество забронированного товара превышает total_quantity.

**Часть 3: Функциональные требования и техническая логика3.1. Управление состояниями (State Machine)**

Инвентарь и Мероприятия имеют жизненный цикл. Правильная обработка статусов гарантирует, что сломанный стул не попадет в заказ, а завершенное мероприятие вернет вещи на склад.**Статусы Мероприятия (Event Status):**

- **Confirmed (Подтвержден):** Реквизит официально забронирован. Эти единицы вычитаются из доступного остатка на выбранные даты.
- **Finished (Завершено):** Мероприятие прошло. Реквизит считается вернувшимся на склад. Бронь снимается, но история сохраняется.

\-----**3.2. Алгоритм расчета доступности (Core Logic)**

Это ключевой раздел для технической главы диплома. Нам нужно определить, сколько единиц товара $X$ свободно в интервале дат $\[D\_{start}, D\_{end}\]$.**Логика пересечения интервалов**

Мероприятие $A$ конфликтует с Мероприятием $B$, если выполняется условие:

$$Start\_A \< End\_B \\quad \\text{и} \\quad End\_A \> Start\_B$$**Алгоритм вычисления остатка:**

1. Система принимает itemId и целевой интервал $\[TargetStart, TargetEnd\]$.
2. Выполняется запрос к БД для поиска всех записей Reservation, связанных с мероприятиями, которые:
   - Имеют статус Confirmed.
   - Пересекаются с целевым интервалом по условию выше.
3. Вычисляется сумма забронированных единиц:
4. $$Q\_{reserved} \= \\sum\_{i=1}^{n} quantity\_i$$
5. Итоговый доступный остаток ($Q\_{available}$):
6. $$Q\_{available} \= Q\_{total} \- Q\_{reserved}$$

**Примечание для разработки:** Эту логику нужно реализовать на уровне базы данных (Prisma/SQL), а не вытягивать все записи в массив JavaScript, чтобы избежать проблем с производительностью при росте базы.-----**3.3. Правила валидации (Business Rules)**

Чтобы данные оставались консистентными, приложение должно соблюдать следующие правила:

1. **Запрет отрицательного остатка:** Система не должна позволять **создавать/обновлять** мероприятие, если хотя бы по одной позиции $Q\_{available} \< Q\_{requested}$.
2. **Защита от редактирования дат:** Если пользователь хочет изменить даты в Confirmed мероприятии, система должна сначала выполнить «виртуальную» проверку. Если на новые даты инвентарь занят — выдается предупреждение, а сохранение блокируется.
3. **Удаление инвентаря:** Запрещено удалять предмет из системы, если он привязан к будущим мероприятиям в статусе Confirmed. Сначала нужно перевести мероприятие в статус **Finished** или удалить его.

\-----**3.4. Обработка исключений и конфликтов**

| Ситуация                               | Реакция системы                                                                                                                                               |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Уменьшение общего кол-ва на складе** | Если админ меняет total_quantity (например, 10 \-\> 5), а на будущее уже забронировано 8 штук — система должна вывести список «конфликтных» мероприятий.      |
| **Наложение «хвостиков»**              | При расчете доступности система должна учитывать время на логистику (например, \+1 день к end_date на возврат и чистку), если это будет выбрано в настройках. |

**Часть 4: Информационная архитектура и данные4.1. Схема базы данных (ER-диаграмма)**

Учитывая количественный учет и твой стек (Prisma \+ PostgreSQL), нам необходима нормализованная структура из трех основных таблиц.**Описание сущностей:**

1. **Item (Инвентарь):** Хранит описание вещей и их общее количество.
2. **Event (Мероприятие):** Хранит метаданные события и его временные рамки.
3. **Reservation (Бронирование):** Связующая таблица (многие-ко-многим), которая определяет, сколько единиц конкретного инвентаря закреплено за конкретным событием.

**Пример схемы Prisma (schema.prisma):**

Фрагмент кода

enum EventStatus {

model Item {

\-----**4.2. Спецификация API (Next.js Route Handlers)**

Все запросы проходят через /api/\*. Взаимодействие строится по принципам REST.**Inventory API**

- **GET /api/inventory**: Получение списка всех предметов.
- **POST /api/inventory**: Создание новой позиции.
- **PATCH /api/inventory/\[id\]**: Обновление данных (например, изменение totalQuantity).
- **DELETE /api/inventory/\[id\]**: Удаление (с проверкой отсутствия активных броней).

**Events API**

- **GET /api/events**: Список всех мероприятий.
- **POST /api/events**: Создание мероприятия (название, даты) **с обязательной проверкой доступности**.
- **GET /api/events/\[id\]**: Полные данные события вместе с вложенным списком reservations.
- **PATCH /api/events/\[id\]**: Обновление статуса или состава инвентаря.

**Availability API (Служебный)**

- **GET /api/availability**:
  - **Параметры:** itemId, startDate, endDate.
  - **Логика:** Возвращает число — сколько единиц данного товара свободно в указанный промежуток. Это критически важный эндпоинт для "живой" валидации на фронтенде.

\-----**4.4. Поток данных (Data Flow)**

1. **Запрос:** Пользователь в интерфейсе меняет дату мероприятия.
2. **Действие:** TanStack Query инициирует фоновый запрос к /api/availability.
3. **Обработка:** Серверный Route Handler обращается к PostgreSQL через Prisma, выполняя агрегацию SUM(quantity) по пересекающимся датам.
4. **Ответ:** Клиент получает число и мгновенно обновляет состояние инпутов (блокирует ввод, если превышен лимит).
